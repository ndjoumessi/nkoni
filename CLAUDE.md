# CLAUDE.md — NKONI

Gestion des cotisations & transparence financière pour associations, familles élargies et
tontines. **SaaS multi-tenant** : chaque organisation cliente a son espace isolé.

Spécifications : `NKONI_Spec_Technique_Dev.md` (mono-tenant d'origine) et
`NKONI_SaaS_Spec_Technique_v1.md` (évolution SaaS). Le code référence ces specs par section
(`§2.2`, `§3.1`, …) — garder cette convention dans les commentaires et commits.

## Structure & stack
- `backend/` — Node 20 + Fastify 5 + Prisma 7 (générateur `prisma-client`, adapter `@prisma/adapter-pg`) + PostgreSQL. TypeScript, tests Vitest.
- `frontend/` — Vite + React + React Router + Tailwind v4. Design system **« Menthe & Encre »** (fintech, dark-only ; tokens oklch **centralisés dans `src/index.css`**, primitives dans `src/components/ui/`). Accent principal **menthe** (jeton `--brass`), or discret en secondaire (`--amber`), rouge/bleu pour les statuts ; fontes **Geist** (UI/titres, Fraunces retiré) + **IBM Plex Mono** pour les montants/données via la classe `.num`. Focus clavier : **un anneau menthe unique** (`:focus-visible` global). CTA primaire = dégradé diagonal émeraude → or. Un thème = édition des VALEURS de jetons (les noms restent stables : `--brass`, `--jade`, `--amber`… gardent leur rôle même après changement de teinte). **Ne pas recopier de valeurs oklch en dur** dans les composants : passer par le TOKEN (`var(--brass|jade|hairline)`) + `color-mix(in oklch, …)` pour l'alpha (cf. `EmptyState`, `RecouvrementHero`, `ErrorState`). **États d'erreur de page** = primitive partagée `ui/ErrorState.tsx` (miroir d'`EmptyState` en tons `--terra`, `role="alert"`, bouton « Réessayer » optionnel) — ne pas redupliquer. `ui/Modal.tsx` embarque un **piège de focus** complet (mémorise le déclencheur, boucle Tab/Shift+Tab, restaure au close). **Logo** : `components/ui/NkoniMark.tsx` = SVG « Cercle d'union » (membres menthe autour de la cagnotte émeraude→or, chef en or), réutilisé partout ; `public/favicon.svg` porte le MÊME design et les **icônes PNG PWA** (`public/pwa-*.png`) en sont dérivées — à **régénérer** via `rsvg-convert` (tuile `#0f1512`) à tout changement de logo, sinon elles restent périmées.

## Commandes
Backend (`cd backend`) :
- `npm run build` — `tsc` (pas de lint séparé côté backend ; le build EST le garde-fou de typage)
- `npm run test -- --run` — Vitest en one-shot. Un seul fichier : `npm run test -- --run tests/membres.route.test.ts` ; un seul cas : ajouter `-t "libellé du test"`.
  - Deux familles de tests : `*.route.test.ts` / `*.service.test.ts` s'exécutent sur **mocks Prisma** (`buildApp({ prisma: mock })` + `app.inject`, aucune DB requise) ; les 6 `*.integration.test.ts` frappent une **vraie Postgres** (`DATABASE_URL` local) et exigent `JWT_ACCESS_SECRET`/`JWT_REFRESH_SECRET` dans `.env` — sans eux, les lancer via `npm run test` les fait échouer (les exclure : `--exclude '**/*.integration.test.ts'`).
- `npm run dev` — serveur tsx watch
- `npx prisma migrate dev` — crée/applique une migration en local (DB `postgresql://nelson@localhost:5432/nkoni`)
- `npm run seed` / `npm run seed:superadmin` — amorçage ADMIN de test / bootstrap SUPER_ADMIN (env `SUPERADMIN_EMAIL`/`SUPERADMIN_PASSWORD`)

Frontend (`cd frontend`) :
- `npm run build` — `tsc -b && vite build`
- `npm run lint` — `oxlint` (doit être clean : 0 finding)
- `npm run test` — **Vitest** one-shot. Deux familles : `*.test.ts` en env **`node`** (client HTTP : refresh-on-401, dédup, déconnexion ; helpers purs) et `*.test.tsx` en env **`jsdom`** (composants React, ex. `DatePicker` en portail) — l'env jsdom est choisi **par fichier** via le docblock `// @vitest-environment jsdom` en tête. `tsconfig.app` **exclut** `*.test.{ts,tsx}` du build ; config `vitest.config.ts` (plugin React pour le JSX, alias `@`).
- `npm run dev` — Vite

Toujours vérifier **build + test** (backend) ou **build + lint (+ test si le client HTTP est touché)** (frontend) avant de présenter un résultat.

**CI (audit C3)** — `.github/workflows/ci.yml` : sur chaque PR + push `main`, job **backend** (Postgres 16 en service → `migrate deploy` + `generate` + `build` + **tous** les tests, *y compris* les `*.integration.test.ts` qui exigent une vraie DB) et job **frontend** (`build` + `lint`). Les invariants critiques (isolation tenant, liens signés) ne dépendent plus de la discipline manuelle — mais la vérif locale avant push reste la règle.

## Architecture — points essentiels

**Isolation multi-tenant (SaaS §2.2)** — défense en profondeur, `backend/src/lib/` :
- `org-context.ts` : `AsyncLocalStorage` portant l'`organisationId` de la requête. `runUnscoped()` = bypass DÉLIBÉRÉ pour les flux sans org (login/refresh/système/seed/super-admin). **Allowlist source (audit archi)** : `tests/runUnscoped-allowlist.test.ts` énumère TOUS les appels `runUnscoped` du `src/` (hors `generated/`) et exige une parité stricte avec une liste justifiée (par fichier + compte) — tout nouvel appel casse le build tant qu'un relecteur ne l'a pas inscrit et justifié (les commentaires ne sont pas un garde-fou exécutable). Pendant côté APPELANTS du test de parité `SCOPED_MODELS ↔ schéma`.
- `tenant-extension.ts` : extension Prisma `$extends` qui injecte `organisationId` sur les 26 `SCOPED_MODELS` (dont `Depense`, `CagnotteEvenement`, `DonCagnotte`, `Amende`). **Fail-closed** : sur un modèle scopé sans contexte org valide → lève `TenantContextError` ; une opération scopée NON couverte par l'extension (ex. `createManyAndReturn`) → lève `OperationNonIsoleeError` (jamais de passe-plat silencieux). Mutation cross-org → `P2025` (→ 404, pas de fuite d'existence).
- `Organisation` n'est **PAS** un modèle scopé (les routes plateforme la lisent sans contexte).
- Ordre des extensions (`lib/prisma.ts`) : audit puis tenant (tenant outermost → scope/fail-close AVANT l'audit).
- **`findUnique[OrThrow]` + `select` : l'extension gère la troncature.** Le contrôle d'appartenance est un POST-filtre sur `res.organisationId`. Si l'appelant fournit un `select` qui **omet** `organisationId`, l'extension l'ajoute le temps de la requête puis le **retire** du résultat rendu (sinon `undefined !== orgId` → faux « introuvable » sur TOUTE ligne, y compris la sienne). Donc un `select` n'a pas à inclure `organisationId` — mais si tu écris un post-filtre analogue ailleurs, souviens-toi que Prisma ne renvoie que les champs sélectionnés. Couvert par `tests/tenant-findunique-select.test.ts` + repro d'intégration.
- **Écritures scopées = FK SCALAIRES (input « unchecked »), jamais la forme relation `{ connect }`.** L'extension force le **scalaire** `organisationId` dans le `data` d'un `create`/`createMany`. Utiliser une relation pour une AUTRE FK (ex. `televersePar: { connect: { id } }`) bascule Prisma en input « checked », où `organisationId` scalaire n'existe pas et la relation `organisation` devient obligatoire → « Argument `organisation` is missing ». Écrire les FK en scalaire (`televerseParId: id`, cast `…UncheckedCreateInput`) comme tous les creates scopés. Régression : `tests/document-create.integration.test.ts`.

**Rôles & permissions** — `backend/src/middlewares/permissions.ts` :
- Matrice `PERMISSIONS[entité][rôle]` + `requirePermission(entité, action)`. Miroir front dans `frontend/src/lib/roles.ts`.
- Entité `Organisation` (paramètres de l'org, **lecture seule**) : ouverte à tous les rôles d'organisation **sauf** MEMBRE_SIMPLE (le quota relève de la gestion). `GET /organisations/moi` (scopé, self-service) renvoie nom/devise/langue défaut/date + le **forfait** courant + **nb de membres ACTIFS vs limite du forfait** (`limiteMembres` = `null` si illimité) + le **chef** (`chefMembreId`/`chefSurnom` + nom/prénom) ; écran `/parametres` (lecture seule, ces paramètres sont **immuables** §5, chef inclus en affichage). Le SUPER_ADMIN reste hors matrice → 403.

**Forfaits (SaaS §3.1)** — enum `Forfait` (`GRATUIT`/`PRO`/`ENTREPRISE`) + champ `Organisation.forfait @default(GRATUIT)` (migration `forfait_organisation`, colonne NOT NULL DEFAULT → backfill implicite). Source de vérité PARTAGÉE `backend/src/lib/forfait.ts` (+ miroir `frontend/src/lib/forfait.ts`) : `limiteMembresForfait(forfait)` → **GRATUIT = 50**, **PRO/ENTREPRISE = `null` (illimité)**. Le quota membres (POST `/membres`, `/membres/import`) lit le forfait de l'org et n'applique le plafond QUE si non `null`. **Attribution = action PLATEFORME manuelle réservée au SUPER_ADMIN** (pas de paiement) : `PATCH /platform/organisations/:id/forfait` (garde `requireSuperAdmin`, `runUnscoped`, 404 sur id inconnu) ; console super-admin = sélecteur de forfait par org (MAJ optimiste). Libellés i18n partagés `commun.forfaits.{GRATUIT,PRO,ENTREPRISE}`.
- **Chef de l'organisation** — SEULE écriture sur `Organisation` : `PATCH /organisations/moi/chef` (désigne/retire un Membre dirigeant + surnom). C'est une **action mutable HORS matrice** (les paramètres §5 restent immuables), gardée par **`requireRoles(['ADMIN','PRESIDENT'])`** (`middlewares/permissions.ts`, factory réutilisable façon `requireSuperAdmin` — à préférer à un `requirePermission('Organisation','update')` qui laisserait croire les paramètres modifiables). Le membre est validé par une lecture **scopée** (`membre.findUnique` → un membre d'une autre org renvoie `null` → 404, isolation tenant) ; l'écriture cible `Organisation` (modèle NON scopé) par id en **FK SCALAIRE** (`chefMembreId`, jamais `{ connect }`). Schéma : FK `onDelete: SetNull` (migration additive `chef_organisation`). Front : désignation depuis la fiche membre (ADMIN/PRESIDENT), badge « Chef » sur liste/fiche, affichage lecture seule sur `/parametres`. `peutDesignerChef` miroir dans `lib/roles.ts`.
- `authenticate` (hook JWT) peuple `req.user` et appelle `orgContext.setOrganisation(req.user.organisationId)`.
- **SUPER_ADMIN** (rôle plateforme transverse, §2.3) : `organisationId` NULL (invariant CHECK en base), **absent de la matrice** → 403 sur toute route tenant. Ses seules routes : `/platform/*` (garde `requireSuperAdmin`). Voir la mémoire `nkoni-super-admin`.

**Auth (§4.5)** — JWT access (Bearer, en mémoire côté front) + refresh (cookie httpOnly, namespace `@fastify/jwt` séparé). Émission factorisée dans `lib/session.ts` (partagée login + auto-inscription). argon2 pour les mots de passe.
- **Révocation de session par époque (audit M5)** — `Utilisateur.sessionEpoch Int @default(0)` (migration `session_epoch`, sans backfill). Le refresh token porte le claim `epoch` = `sessionEpoch` à l'émission (`session.ts`) ; `POST /auth/refresh` rejette (401) si `(payload.epoch ?? 0) !== user.sessionEpoch`. `changerMotDePasse` (self) et `reinitialiserMotDePasse` (admin) **incrémentent** `sessionEpoch` → **toutes les autres sessions tombent au prochain refresh** (TTL access 15 min). **Rétro-compatible** : un refresh token émis AVANT la feature n'a pas de claim `epoch` → traité comme `0`, donc valide tant que l'utilisateur n'a pas changé son mot de passe (pas de déconnexion de masse au déploiement). Le changement de mot de passe réémet une session fraîche à l'appelant.
- **Rotation des refresh tokens + détection de réutilisation (audit M5)** — refresh tokens **STATEFUL** : modèle `RefreshToken` (migration `refresh_token`) `{ jti unique, utilisateurId, familleId, revoke, expiresAt }`, **NON scopé** (pas d'`organisationId` → accès en `runUnscoped`/direct, hors `SCOPED_MODELS`, parité `tests/tenant-scoped-models.test.ts` préservée). Chaque refresh porte `jti` (identité) + `fam` (famille, constante sur toute une chaîne de rotations issue d'un même login) + `rem` (propage le `rememberMe` à travers les rotations — sans quoi un « se souvenir de moi » retomberait à 7 j au 1ᵉʳ refresh). `emettreSession(reply, user, rememberMe, { prisma, familleId? })` **persiste** le token si `prisma` est fourni (login, inscription, refresh, reset MDP) ; `familleId` fourni = **reste dans la même famille** (rotation), sinon neuve. `POST /auth/refresh` : si `jti` présent → lit le `RefreshToken` (absent/expiré → 401), sinon **`revoke === true` = replay d'un token volé → révoque TOUTE la famille (`updateMany`)** + 401 ; sinon **révoque ce `jti` et réémet dans la même famille** (rotation). Un refresh **legacy** (émis avant la feature, **sans `jti`**) est accepté une fois puis **migré** vers stateful (rétro-compat, pas de déconnexion de masse au déploiement). `POST /auth/logout` révoque désormais **la famille côté serveur** (best-effort : cookie effacé même si le refresh est absent/illisible) — déconnexion réelle, plus un simple `clearCookie`. Complète (n'annule pas) la **révocation par époque** ci-dessus : l'époque coupe TOUTES les sessions au changement de MDP, la rotation coupe UNE chaîne compromise sur replay.

**Sécurité HTTP (audit W)** — `app.ts` : **`trustProxy: true` est OBLIGATOIRE** (derrière Railway/Vercel l'IP client réelle est dans `X-Forwarded-For` ; sans ça `@fastify/rate-limit` keyerait sur l'IP du proxy = **un seul seau pour tous → lockout global**). `@fastify/rate-limit` global `max: 300`/`1 minute` (surcharge par route via `config.rateLimit`). `@fastify/helmet` avec `crossOriginResourcePolicy` assoupli (proxy same-origin Vercel). `lib/env.ts` : fail-fast sur les secrets OBLIGATOIRES (`JWT_*`) au démarrage + **warnings prod NON bloquants** (audit W1, muets hors prod) si `CORS_ORIGIN`/`RECU_LINK_SECRET`/`BLOB_READ_WRITE_TOKEN` retombent sur leur défaut risqué.

**Audit trail (V2 §5)** — `lib/audit-middleware.ts`, extension Prisma, 6 entités. Best-effort (n'échoue jamais l'op métier), `passwordHash` toujours exclu, sauté pour les écritures non scopées (`runUnscoped`).

**Documents / Blob (V2 §5)** — `services/document.service.ts` (règles d'accès pures, `chargerParent`/`peutVoirDocument` : visibilité HÉRITÉE du parent polymorphe, jamais dupliquée), `lib/blob.ts` (client Vercel Blob). Le store Vercel Blob est en **PRIVATE** (documents sensibles) : `put({ access: 'private' })` et l'URL n'est **jamais** exposée au client. Téléchargement = **proxy authentifié** `GET /documents/:id/contenu` qui applique `peutVoirDocument` PUIS lit le contenu via `BlobClient.lireContenu` = `@vercel/blob` `get(url, { access: 'private' })` (auth par `BLOB_READ_WRITE_TOKEN`, pas un `fetch` d'URL publique). `BlobClient` (put/del/lireContenu) est mockable → tests sans réseau ; le round-trip réel contre le store n'est vérifiable qu'en prod.

**Scheduler notifications** — `services/notification-scheduler.ts` : `node-cron` **in-process** (03:00 Africa/Douala, un seul process Railway long-vivant). Génère les `COTISATION_RETARD` en **bouclant org par org**, chaque itération enveloppée dans `orgContext.run({ organisationId })` (jamais un `runUnscoped` global qui mélangerait les tenants). Cœur métier découplé du cron (`anneeCourante` + `now` injectés) → testable sans horloge réelle.

**Dashboard (§ dashboard)** — `services/dashboard.service.ts` : cœur PUR (`anneeCourante`/`moisCourant` injectés → testable sans horloge) exposé par `routes/dashboard.route.ts` avec une **vue par rôle** (COMPLET / FINANCIER / RESTREINT). Trois conventions non-évidentes, verrouillées par une revue : (1) **Enrichissement best-effort** — un agrégat OPTIONNEL greffé sur la vue de base (ex. `financesConsolidees` = solde caisse + cagnottes ouvertes + amendes, réservé aux rôles argent/dirigeant ADMIN/PRESIDENT/TRESORIERE/COMMISSAIRE_COMPTES) passe par `.catch(() => undefined)` DANS le `Promise.all` : s'il échoue, la carte se **masque** (champ optionnel côté front) au lieu de faire échouer TOUT le `GET /dashboard`. (2) **« Mois/jour courant applicatif » = `Africa/Douala`** (comme le scheduler), **JAMAIS `getUTCMonth()`** — dériver la date courante en UTC affiche le mois précédent la 1ʳᵉ heure locale du mois (bug vécu sur « anniversaires du mois ») ; le mois de naissance d'un MEMBRE, lui, se lit en UTC (une date de naissance ne glisse pas d'un fuseau). (3) **Comparaison N vs N-1** (courbe mensuelle) **bornée au mois courant** côté front (`if (mois <= moisCourant)`, cumul avec `?? 0` anti-NaN) : sinon la N-1 cumule toute l'année écoulée face à une année courante figée au mois en cours → faux « retard » sur les mois à venir. **Relance WhatsApp** (dashboard + fiche membre) : lien `wa.me` construit UNIQUEMENT via `lib/utils.ts::lienRelanceWhatsApp` (numéro E.164 `telephoneWaMe` + `encodeURIComponent`, `null` si invalide → bouton masqué) — ne pas reconstruire l'URL à la main ; le bouton reste gardé par `peutGererMembres` (jamais MEMBRE_SIMPLE, pas d'auto-relance).

**Reçus PDF & WhatsApp (§4.6)** — `services/recu-pdf.service.ts` (génère le PDF en RÉUTILISANT `export-style.ts`, stocke sur Blob PRIVATE, idempotent) ; téléchargement = **proxy authentifié** `GET /recus/:id/pdf` (comme les Documents). Envoi WhatsApp = `services/whatsapp.service.ts` : client **mockable** (`WhatsAppClient`), `vraiWhatsAppClient` = Meta Cloud API, **no-op sans** `WHATSAPP_TOKEN`/`WHATSAPP_PHONE_ID`. `envoyerRecuWhatsApp` est **best-effort** (ne lève jamais) et respecte `notificationsActives` du destinataire. **Téléphones normalisés E.164** (`lib/telephone.ts::normaliserTelephone`, défaut Cameroun `237`, indicatif paramétrable) AVANT l'appel client : local `6XXXXXXXX → 2376XXXXXXXX`, nettoyage espaces/`+`/`00`, invalide → pas d'envoi. **Templates pré-approuvés Meta NON implémentés** (reporté tant que le compte Meta Business n'existe pas).

**Liens PUBLICS signés (§4.6/§4.7)** — `lib/recu-lien.ts` : signe l'accès public NON authentifié via **HMAC-SHA256** (`RECU_LINK_SECRET`, **repli `JWT_ACCESS_SECRET`** — le poser distinct sur Railway = révocation des liens sans invalider les sessions ; cf. RUNBOOK/§ env). Deux usages, **préfixes DISTINCTS** (séparation de domaine, une signature ne vaut jamais d'un domaine à l'autre) : `recu-pdf-public:v1:` → `GET /recus/:id/pdf-public?t=<sig>` (télécharge SON reçu sans compte), `carte-statut:v1:` → `GET /membres/:id/statut-public?t=<sig>` (page HTML de vérif ouverte par le QR de carte). **Recette d'une route publique signée** : (1) `verifierSignature*` en **temps constant** (`timingSafeEqual` + garde de longueur) AVANT tout accès DB → **`404` uniforme** sur signature invalide / id inconnu (pas d'énumération ni de fuite d'existence) ; (2) résoudre l'org du reçu/membre **`orgContext.runUnscoped(async () => await prisma…)`** — **`await` OBLIGATOIRE DANS le callback** : une `PrismaPromise` est PARESSEUSE, l'awaiter dehors exécute la requête HORS contexte → l'extension tenant fail-close (500) ; (3) générer/rendre **DANS `orgContext.run({ organisationId })`**. La `signaturePartage`/l'URL signée n'est renvoyée QUE par les endpoints **authentifiés** (jamais publique tant que non partagée). Page publique = **AUCUN montant**, `esc()` sur toute donnée, `noindex`. Régression verrouillée par `tests/cartes-statut-public.integration.test.ts` (vraie Postgres).

**Photo du membre (§4.11)** — `Membre.photoBlobUrl`/`photoMime` (nullables) : **même pattern que Documents/Reçus** (Blob PRIVATE + **proxy authentifié** `GET /membres/:id/photo`, URL jamais exposée). `routes/membre-photo.route.ts` (POST/GET/DELETE) : upload **multipart JPEG/PNG ≤ 5 Mo** (allowlist MIME ; plafond `@fastify/multipart` 11 Mo/1 fichier borne le DoS), MEMBRE_SIMPLE = sa propre photo. Rendue sur la carte PDF (avatar, repli initiales) mais **JAMAIS sur la page publique de statut** (PII).

**Trésorerie / Dépenses (§5)** — modèle `Depense` **scopé** (l'un des 26 SCOPED_MODELS ; parité schéma↔Set garantie par `tests/tenant-scoped-models.test.ts`) + `routes/depenses.route.ts` : workflow `BROUILLON → EN_ATTENTE → APPROUVEE|REJETEE → PAYEE` (`services/tresorerie.service.ts::validerTransition`/`estEditable`, purs). Gardes de rôle distinctes : approbation = ADMIN/PRESIDENT/COMMISSAIRE_COMPTES, paiement = ADMIN/PRESIDENT/TRESORIERE. `GET /tresorerie` = solde de caisse (Σ versements − Σ dépenses APPROUVEE/PAYEE) + ventilation par catégorie. **Pattern « séparation gestion/argent » réutilisé** par Cagnottes (§4.9) et Amendes (§4.10) : le CRUD d'entité passe par la matrice `requirePermission`, mais les **flux d'argent** (dons/reversement, encaissement/annulation) sont gardés à part dans la route par `requireRoles(['ADMIN','PRESIDENT','TRESORIERE'])`.
- **Invariant financier des versements en service (audit M1/M2)** — `services/versement.service.ts` porte le cœur (crée le versement ET incrémente `montantVerse`/`montantValorise` dans une même tx §5) ; `routes/versements.route.ts` **délègue** (la route ne recalcule plus l'invariant à la main). Filet de sécurité : `GET /tresorerie/reconciliation` (`reconcilierVersements`) compare, membre par membre, la somme des versements aux cumuls stockés et renvoie les **écarts** — endpoint auth-gated de contrôle, pas d'écriture.
- **Pagination par offset (audit m4)** — helper PARTAGÉ `backend/src/lib/pagination.ts` (`resoudrePagination` borne page/pageSize : défaut 25, max 100 ; `PAGINATION_PROPS` = fragment ajv à fusionner dans la querystring ; réponse stable `PageResultat<T> = { items, total, page, pageSize }`) pour les listes qui grandissent sans limite (dépenses paginées d'abord). Miroir front `frontend/src/components/ui/Pagination.tsx` + `Paginated<T>` dans `lib/api.ts` — **réutiliser, ne pas réinventer** un schéma de pagination ad hoc.
  - **`/membres/statuts` = réponse BORNÉE, pas offset (audit m4)** — le statut de cotisation est **calculé** (borne d'années × barèmes, pas une colonne DB), donc une vraie pagination offset exigerait de matérialiser le statut (reporté, aucun besoin). À la place `calculerStatutsMembres(prisma, annee, where?, limite?)` (`services/membreStatut.service.ts`) renvoie `{ items, total, tronque }` et **borne à `PLAFOND_STATUTS_MEMBRES = 1000`** ; `tronque = total > limite`. Côté front, `membresApi.listStatutsPage` expose l'objet (bannière de troncature sur `MembresPage`) tandis que `membresApi.listStatuts` **déballe `.items`** pour les nombreux appelants « tableau seul » (sélecteurs, dashboard, ⌘K). Les consommateurs de `calculerStatutsMembres` (cartes, relevé) déstructurent `.items`.

**Import de membres (§5.2) — parsing côté serveur (audit m6)** — le fichier téléversé (.xlsx / .csv) est parsé **sur le backend** par `services/import-parse.service.ts` (`parserFichierImport` : `exceljs` — déjà utilisé pour les exports — pour le XLSX, parseur CSV minimal, détection par extension) via `POST /membres/import/fichier` → `{ entetes, lignes }` pour l'étape de mapping. La dépendance **`xlsx` a été RETIRÉE du frontend** (CVEs connues + ~400 KiB de bundle en moins) : ne pas la réintroduire côté client. L'étape de commit reste `POST /membres/import` (idempotence + quota forfait, cf. [[nkoni-super-admin]] et note Forfaits).

**PWA hors-ligne & idempotence (§ PWA)** — `vite-plugin-pwa` (Workbox `generateSW`) : SW `autoUpdate`, GET `/api/*` (hors `/auth`) en **NetworkFirst**, jamais de POST/`/auth` en cache, `png` dans `globPatterns` (icônes précachées). Écriture hors-ligne = file **IndexedDB** (`lib/offline-queue.ts`) + rejeu optimiste (`lib/offline-sync.ts`, `hooks/useSyncHorsLigne.ts`) avec **clé d'idempotence** cliente (uuid). Côté backend, `Versement` et `Membre` portent `idempotenceKey` + unique `(organisationId, idempotenceKey)` ; les routes POST lisent l'en-tête `Idempotence-Key`, renvoient l'existant sur rejeu. **Gotcha P2002** : le re-fetch idempotent n'a lieu QUE si `estConflitIdempotence(err)` (`lib/idempotence.ts`) confirme via `err.meta.target` que la contrainte violée est bien `(organisationId, idempotenceKey)` — un P2002 sur une AUTRE contrainte doit être RELEVÉ (sinon on re-fetch la mauvaise ligne / null).

**Internationalisation FR/EN (§4)** — interface + messages serveur + notifications + **formats** dates/montants intégralement traduits (lots F1–F6 mergés sur `main`). Préférence de langue **par utilisateur** (`Utilisateur.langue Langue?`, nullable ; indépendante de `Organisation.langueDefaut`), fixée via `PATCH /auth/me/langue` (self-service, réémet un access token portant la langue).
- **Catalogues = fragments par namespace, agrégés par un index, parité vérifiée à la compilation** (même recette des 2 côtés) :
  - Backend `backend/src/locales/{fr,en}/<domaine>.ts` (chacun `export const messages = {…}`), agrégés par `index.ts` ; `en/index` typé `Messages` (dérivé de FR). Clés **plates** `<domaine>.<message>`, interpolation `{nom}`. `t(langue, clé, params?)` dans `lib/i18n.ts`.
  - Frontend `frontend/src/locales/{fr,en}/<ns>.ts` (chacun `export default { <ns>: { … } }`, clés **imbriquées**), agrégés par `index.ts` ; `en/index` typé `Catalogue`. `react-i18next` (namespace unique `translation`), interpolation `{{var}}`, pluriels `_one`/`_other`.
  - **Ajouter un namespace = créer un fragment `fr/<x>` + `en/<x>` et l'importer/spreader dans les DEUX `index.ts`.** Namespaces disjoints → un fragment par module, jamais d'écriture partagée (ce qui a permis de paralléliser les lots). Ne PAS traduire dans un fichier partagé sans le signaler.
- **Langue effective** (`langueEffective(user)` dans `auth.service.ts`) : `Utilisateur.langue` **sinon** `organisation.langueDefaut` (un nouvel utilisateur hérite du défaut de son org, cohérent avec l'inscription ; null seulement pour le SUPER_ADMIN sans org). Elle est **résolue à l'émission du token** (`session.ts`) et portée par l'access token — et renvoyée par `/auth/me`, login, inscription. `langueDeRequete(req)` résout : `req.user.langue` (du token) → en-tête `Accept-Language` → FR. Sans `req` sous la main, passer `reply.request`.
- **Traduction des erreurs = à la frontière HTTP, par TYPE d'erreur** : les services restent **i18n-agnostiques** (ils lèvent des erreurs typées portant les *données* — ex. `EmailDejaUtiliseError.email` — jamais la langue) ; la route mappe `err instanceof XError → t(langue, 'clé', { …données })`. Ne PAS renvoyer `err.message` (français figé) au client.
- **Notifications in-app** (`notification.service.ts` / `notification-scheduler.ts`) : rendues dans la langue du **DESTINATAIRE** (le membre concerné), jamais de l'acteur qui déclenche l'action — `resoudreLangueDestinataire(prisma, destinataireId)` = langue perso ↩ défaut org ↩ FR. Namespace `notifications.*`.
- **Frontend runtime** : codes backend `FR`/`EN` ↔ i18next `fr`/`en` via `versI18n`/`versBackend` (`lib/i18n.ts`). La préférence serveur est appliquée par `AuthContext` (`appliquerLangue`) au login/inscription/réhydratation et **prime** sur le `localStorage` (repli hors-ligne : navigateur → FR). Sélecteur dans **Mon profil**. Convention composant : `const { t } = useTranslation()`, libellés résolus dans le composant (pas de map de libellés figés au niveau module). **`t()` est TYPÉ contre le catalogue** (augmentation `declare module 'i18next'` dans `src/react-i18next.d.ts` : `resources: { translation: Catalogue }`) → une clé statique inexistante est une **erreur de compilation**. Pour une clé **dynamique** (indexée par une variable/valeur d'énum, ex. `` `amendes.statuts.${statut}` ``), l'envelopper dans **`cleI18n(...)`** (`lib/i18n.ts`, identité au runtime) qui contourne le contrôle strict.
- **Formats locale-aware (§4/§5, lot F6 — FAIT)** : dates et montants suivent la locale/devise, plus de `fr-FR`/FCFA en dur.
  - **Montants** : `formatMontant()` (`frontend/src/lib/format.ts` ; miroir backend dans `lib/i18n.ts` pour les notifications) rend `Intl.NumberFormat({ style:'currency' })` dans la LANGUE d'interface + la DEVISE de l'org. `FCFA` n'étant pas un code ISO 4217, il est mappé sur **`XAF`** (dont `Intl` restitue justement « FCFA » en fr) — sans ce mappage, `Intl` lèverait `RangeError`. Sans décimales (montants stockés en entiers).
  - **Devise côté front** : `Organisation.devise` descend via `/auth/me` + login + inscription (`AuthUser.devise`) et est appliquée par `AuthContext` (`appliquerDevise`, miroir de `appliquerLangue`) au login/inscription/réhydratation ; défaut FCFA, réinitialisée au logout. Côté backend, une notif est rendue dans la devise du DESTINATAIRE (`resoudreDeviseDestinataire`, comme la langue).
  - **Dates** : `formatDate()`/`formatDateHeure()` (`lib/utils.ts`) formatent selon la langue courante (i18next) ; les composants passent leurs options ponctuelles (numérique court, mois long…). Miroir backend `formatDateHeure(date, langue)` dans `lib/i18n.ts` (pour les exports — plus d'ISO brut dans un document).
- **Exports PDF/Excel (§5.9 + Rapports)** — contributions (`services/export.service.ts`) ET rapports Évolution/Comparaison/Multi (`services/export-rapport.service.ts`) partagent le **style premium « Menthe & Encre »** de `services/export-style.ts` : palette oklch→sRGB **assombrie pour l'impression** sur blanc, `enteteDocument` + `dessinerCorpsPremium` (tableau PDF générique : titre menthe, filet or, bandeau d'en-tête, zébrure, ligne TOTAL — réaffiche le bandeau si le TOTAL déborde en page 2 ; méta bornée à 1 ligne), stylers Excel (bandeau menthe, zébrure, montants `#,##0` à droite — valeurs restées NUMÉRIQUES). **Réutiliser, ne PAS redupliquer.** **Injection de formule tableur (audit Sécu E2)** : tout TEXTE d'origine utilisateur écrit en cellule `.xlsx` passe par `export-style.ts::neutraliserFormuleCellule` (préfixe `'` si la valeur commence par `= + - @ \t \r` → jamais évaluée à l'ouverture Excel/LibreOffice/Sheets). Appliqué aux `nom`/`prenom` de l'export contributions ; `export-rapport.service.ts` n'écrit que des libellés CONSTANTS + des nombres (aucun vecteur). Sans effet en PDF. **Reste ouvert** : le CSV super-admin côté front (`SuperAdminPage.tsx::celluleCsv`) — même neutralisation à porter. Langue+devise via `lib/export-locale.ts::resoudreLocaleExport(req, prisma, avecDevise)` — devise = requête DB (`resoudreDeviseDestinataire`) **uniquement si PDF** (l'Excel garde des nombres, pas de requête pour le `.xlsx` par défaut). **Gotcha PDFKit** : la police Helvetica intégrée (WinAnsi) n'encode PAS l'espace fine insécable d'`Intl` (U+202F, séparateur de milliers → rendue « / ») ni le `Δ` grec → passer les montants par `montantExport`/`nombreExport` (normalisent U+202F/U+00A0) et éviter tout caractère hors Latin-1 dans le PDF. `nombreExport` (sans devise) pour les tableaux denses (multi-années) où « FCFA » déborderait ; la devise est alors rappelée dans le sous-titre. **Variations en %** (PDF de comparaison) : `pourcentExport(n, langue)` — % signé (`+` explicite ; `-` du formatage), localisé, décimales conservées, U+202F/U+00A0 + signe moins U+2212 → ASCII. La **sémantique de variation** est centralisée dans `rapport.service.ts::variationPourcent` (type `Variation = number | 'nouveau' | null`) : `'nouveau'` = apparition (base 0 → positif, libellé « Nouveau »/« New »), `0` = resté à zéro (« 0 % »), `null` = incomparable / année sans barème (« n/a »).
- **Hors périmètre** : le contenu saisi par les utilisateurs (texte d'une Résolution, nom d'un Membre…) n'est JAMAIS traduit. `lib/roles.ts` n'expose plus de libellés (le code mort `ROLES.label`/`libelleRole` a été retiré) — les rôles sont rendus via `t('utilisateurs.roles.<r>')`, `ROLES` n'est plus qu'un `readonly string[]` de valeurs.
- **Gotcha migration** : après `prisma migrate dev`, **relancer `npx prisma generate`** — le client généré (`src/generated/`, gitignoré) peut rester obsolète et faire échouer les tests d'intégration réels (« Unknown field ») alors que le build TS passe.

**Frontend — architecture applicative** (`frontend/src/`) :
- **Routing** (`App.tsx`) : `react-router-dom` déclaratif. Trois zones : pages publiques (`/`, `/login`, `/inscription`), console **plateforme** `/super-admin` (garde `SuperAdminRoute`, layout autonome **hors** `AppShell`), et pages tenant sous `<ProtectedLayout>` = `ProtectedRoute` (garde d'auth) + `AppShell` (coquille + nav) via `<Outlet/>`. **Routes statiques déclarées avant les paramétrées** (`/membres/nouveau` avant `/membres/:id`).
- **Session** (`contexts/AuthContext.tsx` + `auth-context.ts`) : l'access token vit **uniquement en mémoire React** (jamais en `localStorage`). La persistance entre reloads repose sur le cookie httpOnly du refresh — au montage, `AuthContext` tente un `/auth/refresh` silencieux puis `/auth/me` pour réhydrater `user`. `useAuth()` expose `login`/`inscription`/`logout`/`changerLangue` + `user`/`accessToken`/`isAuthenticated`/`loading`. Il applique la langue (`appliquerLangue`, §4) **et la devise** (`appliquerDevise`, §5/F6) du serveur, enregistre le **pont d'auth** du client HTTP (`configurerAuthBridge` : token rafraîchi ↔ setState, session expirée ↔ logout) et arme le refresh proactif.
- **Client HTTP** (`lib/api.ts`) : `fetch` minimal, **`credentials: 'include'` obligatoire** (envoi du cookie refresh). Le token est passé explicitement en `Bearer` par appel. **Refresh-on-401** : au 1er 401 d'une requête authentifiée, `request()` appelle `/auth/refresh` (**dédupliqué** — une seule promesse en vol pour N requêtes concurrentes), remplace le token (`onTokenRefreshed` → `AuthContext`) et **rejoue la requête UNE fois** (`permettreRetry` anti-boucle) ; si le refresh échoue → `onSessionExpired` vide la session → `ProtectedRoute` redirige vers `/login`. `AuthContext` arme aussi un **refresh proactif** ~60 s avant l'expiration de l'access token (TTL 15 min). `ApiError` porte le statut HTTP ; `messageErreur(e)` distingue réponse d'erreur serveur vs rejet `fetch` (réseau/CORS). Base : `VITE_API_URL` (défaut `http://localhost:3000`) — en prod, proxy same-origin Vercel (`/api/*`). **Découpage par domaine (audit archi)** : `lib/api.ts` n'est plus qu'un **barrel** (`export * from './api/*'`) ; le cœur HTTP (`request`, `ApiError`, `messageErreur`, refresh-on-401, `configurerAuthBridge`) vit dans `lib/api/core.ts` (`request` EXPORTÉ pour les modules), les types transverses (`Paginated<T>`, `StatutContribution`, `ModeVersement`…) dans `lib/api/types.ts`, et **un module par domaine** `lib/api/<domaine>.ts` (membres, cagnottes, amendes, rapports…) exportant son objet `<domaine>Api` + ses types. **Tous les imports restent `from '@/lib/api'`** (le barrel préserve chaque nom exporté) — ne pas importer un sous-module directement. `tsc -b` garantit l'absence d'export dupliqué à travers le barrel.
- **Pages & data-fetching** : une page par écran dans `pages/` (liste/détail/formulaire par entité). Chargement de données **dans la page** (`useState`/`useEffect` + fonctions de `api.ts`) ; peu de hooks partagés (`hooks/useDashboard.ts`). Miroir des permissions dans `lib/roles.ts` pour masquer/afficher les actions selon le rôle (source de vérité = matrice backend).
- **Popovers/overlays ancrés à un déclencheur** : **rendus en portail** (`createPortal` dans `<body>`) + `position: fixed` calculée depuis le `getBoundingClientRect()` du déclencheur (recalcul au scroll capture + resize + changement de vue). **Raison structurelle** : `nk-reveal` anime en `forwards` → laisse un `transform` résiduel → chaque bloc animé devient un **contexte d'empilement** ; un popover simplement `absolute` (même `z-40`) reste piégé dans son bloc parent et se fait **recouvrir** par le bloc frère suivant (autre `nk-reveal`/transform/z-index) → illisible ET non cliquable (bugs vécus `/audit` puis `/fonctions`). Le portail l'immunise structurellement.
  - **Primitives PARTAGÉES** (ne PAS redupliquer) : `components/ui/usePopoverFlottant.tsx` (portail + positionnement `fixed` avec bascule verticale/bornage viewport + clic-extérieur épargnant le portail + Échap `defaultPrevented`-safe → `rendreFlottant(children, {className, 'aria-label'})`) et `components/ui/GrilleAnnees.tsx` (grille d'années par décennie : nav décennie ‹/›, cellules hors `[min,max]` désactivées, clavier, **contrôlée** — le parent garde `focusAnnee`/le focus roving/`conserverFocusRef`/l'annonce aria-live). Réutilisées par **`DatePicker`** (calendrier + sélection rapide mois/année : en-tête cliquable → grille années → grille mois) ET **`SelecteurAnnee`** (année seule, ex. barème ; Entrée sur le déclencheur soumet le formulaire — parité `<input>`). **`components/membres/SelecteurMembreUnique.tsx`** = sélecteur d'UN membre **cherchable** (remplace un `<select>` natif ingérable à 30+ membres ; `usePopoverFlottant` + `normaliserTexte` recherche sans casse/accents ; **navigation clavier combobox APG** : `role="combobox"` + `aria-activedescendant`/`aria-controls`, flèches ↑/↓ + Entrée, surbrillance + `scrollIntoView`, ids via `useId` ; prop `optionTous` → mode filtre) — réutilisé Amendes + Cagnottes (bénéficiaire/don), à préférer à tout nouveau `<select>` de membre.
  - **Piège de focus + verrou de scroll** : `Modal.tsx` l'embarque déjà ; pour les autres overlays `aria-modal` **sans** Modal (CommandPalette ⌘K, drawer mobile de l'`AppShell`), utiliser le hook PARTAGÉ `hooks/useFocusTrap.ts` (`useFocusTrap(ref, active)` : mémorise le déclencheur + verrouille le scroll body, boucle Tab/Shift+Tab dans `ref`, restaure au close ; **ne force PAS le focus initial** — le composant le pose lui-même ; Échap/clic-extérieur restent gérés par l'appelant). Reprend la logique de `Modal` **sans** la refactorer — ne pas rustiner un trap ad hoc.
  - **Focus roving** : les flèches ‹/› d'en-tête posent `conserverFocusRef` pour que l'effet de focus ne les vole PAS vers une cellule (sinon l'activation clavier répétée casse). **aria-live** : région `sr-only` dédiée (un `aria-label` sur le bouton d'en-tête masquerait l'annonce du changement).
  - Échelle z : contenu/nav `z-40` · popover portail & `Modal` `z-50` · `Toast`/`CommandPalette` `z-[100]`. **Ne PAS** rustiner un recouvrement au cas par cas avec des `z-index` sur les cartes. Tests composant sous **jsdom** (`*.test.tsx`, docblock `// @vitest-environment jsdom`) ; `partage-popover.test.ts` prouve au niveau source que les deux composants importent bien les primitives (pas de copier-coller).

## Conventions
- **Français** partout : noms de fonctions métier, commentaires, messages d'erreur, commits.
- **Branches de feature** (`feat/...`), jamais de commit direct sur `main`. Merge `--no-ff`. Le PO exécute lui-même les commandes prod ; guider et vérifier.
- **Migrations réversibles** : une valeur d'enum Postgres ne peut pas être utilisée dans la transaction qui l'ajoute → séparer `ADD VALUE` et son usage (CHECK/backfill) en deux migrations.
- `backend/src/generated/` est **gitignoré** (régénéré au déploiement via `postinstall: prisma generate`).
- Nouveau flux sans requête HTTP (script, scheduler, seed) → l'envelopper dans `orgContext.run({ organisationId })` ou `runUnscoped()`, sinon fail-close.

## Déploiement
- **Backend → Railway** : `startCommand = npx prisma migrate deploy && npm run start` (migrations auto-appliquées au déploiement). `DATABASE_URL` = référence `${{ Postgres.DATABASE_URL }}`. `$PORT` injecté. **Le `&&` conditionne le boot** : si `migrate deploy` échoue (ex. `DATABASE_URL` invalide → `P1000`), `start` ne tourne pas, le healthcheck `/health` échoue, le déploiement passe **FAILED**, et Railway **continue de servir le DERNIER déploiement réussi**.
- **Watch Path `/backend/**` (Railway) — un push qui ne touche QUE `frontend/` ne déploie PAS le backend.** `watchPatterns: ["/backend/**"]` est versionné dans `backend/railway.json` **et** posé au dashboard (Root Directory du service = `/backend`, mais les watch patterns restent **relatifs à la racine du dépôt** — cf. `configFile: /backend/railway.json` stocké repo-root-relatif). Un merge front-only (ou doc-only) apparaît alors **`SKIPPED`** dans `railway deployment list` : c'est **normal**, pas un échec. Un changement sous `backend/` (dont `backend/prisma/` = migrations) déclenche bien un déploiement. Le Root Directory seul NE filtre PAS (sans watch pattern, tout push `main` déployait). Corollaire : **ne vérifier Railway que quand `backend/` a changé** ; pour un push front-only, vérifier Vercel.
- **Vérifier un déploiement = statut RÉEL, pas `/health`.** Un `/health` à 200 peut être servi par une ancienne image alors que le dernier déploiement a échoué. Utiliser le CLI `railway` (authentifié, projet `nkoni-backend`) : `railway deployment list` (le plus récent doit être **SUCCESS**, pas `BUILDING`/`FAILED`/`SKIPPED` — voir Watch Path) + `railway status` (le **deployment ID actif** doit être celui du push ; le précédent bascule `REMOVED`) ; logs d'échec via `railway logs --build <id>` (build) et `railway logs --deployment <id>` (start/migrate).
- **Frontend → Vercel** : racine projet = `frontend/`. `vercel.json` proxifie `/api/*` vers le backend Railway (**same-origin** → le cookie refresh fonctionne) + fallback SPA. Prod : `nkoni.vercel.app`.
- Variables backend : `JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET`, `JWT_ACCESS_TTL`, `JWT_REFRESH_TTL`, `CORS_ORIGIN` (liste séparée par virgules), `REFRESH_COOKIE_NAME`/`_PATH`, `BLOB_READ_WRITE_TOKEN`, `NODE_ENV`.
